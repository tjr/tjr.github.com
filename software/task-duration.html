The Big Picture

Lethargy runs rampant in today's society. Most people have lots of things they would like to be doing, but instead of getting those things done, they find themselves watching reruns of Laverne & Shirley in front of a giant flat-screen television, the expense of which is racking up interest payments on their credit cards. Even when someone has the realization that they really ought to just turn the television set off and get to work on one of the many goals they ostensibly have, the thought of completing a whole project can seem insurmountable.

Enter the task list. This handy method of breaking a huge project up into smaller, more manageable pieces has been the subject of many self-help books and implemented in a plethora of software tools. One feature that tends to only be in high-end, cumbersome project management tools is the notion of how long a task is expected to take. It might be easier to turn off the television and saunter over to your desk if you knew that you could make real, tangible progress on writing that novel / learning Italian / fixing the cabinet / whatever by completing a five minute task. Just five minutes, and you've accomplished something. After five minutes, you might feel on a roll and knock of a fifteen minute task, or an hour task. Or you might go back to watching television. But either way, you got something done, and you feel better about it.

The goal of this project is to create a task management system that highlights the expected duration of tasks, in hopes of inspiring more people to make progress on things that really matter in their lives. And if you happen to be a professional television critic, then by all means, enter a 22-minute task for watching Laverne & Shirley.

Design Summary

This software is developed using Ruby on Rails, run off of a single Slicehost virtual server; if user requirements demand, machine resources can be increased and/or the foundational architecture modified.

The data model needs to consist of roughly:

  • Users Table
    • Email address
    • Password
    • Real name
    • User name (optional, if the user wants a nice URL for public tasklist viewing)
    • Public or private (most users will probably want to keep their task list private, but some may want to share it)
  • Categories Table
    • Category name
    • Created by (which user?)
  • Projects Table
    • Project name
    • Project description (optional)
    • Created by (which user?)
    • Category (member of which?)
  • Tasks Table
    • Task summary
    • Task details (optional)
    • Estimated duration
    • Created by (which user?)
    • Status (planned, in-progress, done, cancelled)
    • Project (member of which?)

Based on that structure, the idea is that a user would create general categories (Personal, Work, City Council, etc.), then create projects within those categories (Learn Italian, Build Autopilot Software, Repair City Government Building, etc.), and then create tasks within those projects (Memorize five new nouns, Hold review for program submodule, Get quote from roofing contractor, etc.).

Page requirements should be minimal. There will be a general home landing page from which users can login / register. Upon logging in, the user is directed to their own task list page, where they can access various views and sorts of their tasks, as well as create new tasks, and modify or delete existing tasks. Each user also has an accounts page, from which they can change their name / email / etc., specify if they want their task list to be publicly visible or not, and totally delete their account. (Unlike with online discussion forums, there is really no functional reason to not allow users to erase everything they've done on the site and go home.)

Privacy and Security

User passwords will be stored encrypted. No secure server is planned at this time.

In theory, whoever has access to the raw database has access to all of the user data stored in it. In practice, the site administrators really don't care what is in the user task lists, and do not browse or search them. If a user requests support that involves cleaning up data in their account, the site administrators would ostensibly some user data, but do not peruse data beyond what is needful to fix problems, and maintain user privacy.

The site administrators must comply with any valid legal or government requests for data, lest they be violating the law.

Data Backups

Once the system is in production use, backups will be made at least daily. Details to be determined later.
